0.2 Resumen

   TaskFlow es una aplicación web de gestión de tareas desarrollada bajo
   el patrón MVC con Laravel 12 y MongoDB como base de datos NoSQL. El
   sistema implementa un CRUD completo para tareas con campos de título,
   descripción, fecha, hora y estado, complementado por un motor de
   notificaciones inteligentes que genera alertas automáticas basadas en
   la temporalidad de las tareas (próximas a vencer en 72 horas o
   vencidas) y eventos inmediatos ante acciones del usuario (creación,
   edición, completada, eliminación). La arquitectura combina backend en
   PHP (Laravel) con herramientas de compilación modernas en Node.js
   (Vite, Tailwind CSS), resultando en una pila híbrida que
   mantiene la productividad de Laravel con la flexibilidad de MongoDB.

   El modelo de datos consta de tres colecciones principales: usuarios,
   tareas y notificaciones, relacionadas mediante referencias ObjectId
   (user_id, task_id) sin necesidad de migraciones gracias a la
   naturaleza schemaless de MongoDB. El controlador principal
   (TaskController) centraliza la lógica de negocio del CRUD de tareas
   y la sincronización de estados de notificación automática mediante
   una máquina de estados que evalúa cada tarea en cada carga del
   tablero, controlando duplicados con campos de seguimiento
   (last_state_notified, notification_dismissed_at). El frontend
   utiliza plantillas Blade con un diseño clean profesional (fondo
   gris claro, tarjetas blancas, tipografía Poppins), barra lateral
   colapsable, modales para formularios, sistema de toasts y una
   página de resumen con estadísticas y calendario, todo estilizado
   con CSS personalizado y Tailwind.

   La aplicación incluye autenticación completa (registro, inicio de
   sesión, recuperación de contraseña, verificación de correo
   electrónico) mediante Laravel Breeze, protección CSRF, middleware
   de autorización y gestión de perfil de usuario. No implementa API
   REST, colas, roles ni pruebas automatizadas, optando por un enfoque
   server-rendered simple y directo. El flujo de datos sigue el ciclo
   estándar: petición HTTP → controlador → validación → modelo
   (MongoDB) → notificación si aplica → respuesta (vista Blade o
   redirección). El proyecto está listo para desplegar con scripts de
   configuración y desarrollo integrados en Composer.

0.2b Abstract

   TaskFlow is a web-based task management application developed using
   the MVC pattern with Laravel 12 and MongoDB as its NoSQL database.
   The system implements a complete CRUD functionality for tasks,
   including fields for title, description, date, time, and status. It
   is complemented by a smart notification engine that generates
   automatic alerts based on task timelines (upcoming tasks within 72
   hours or past due) and immediate events triggered by user actions
   (creation, editing, completion, deletion). The architecture combines
   a PHP backend (Laravel) with modern Node.js build tools (Vite,
   Tailwind CSS), resulting in a hybrid stack that maintains
   Laravel's productivity with MongoDB's flexibility.

   The data model consists of three main collections: users, tasks, and
   notifications, linked by ObjectId references (user_id, task_id)
   without requiring migrations thanks to MongoDB's schemaless nature.
   The main controller (TaskController) centralizes the business logic
   for task CRUD operations and automatic notification status
   synchronization using a state machine that evaluates each task on
   every dashboard load, controlling duplicates with tracking fields
   (last_state_notified, notification_dismissed_at). The frontend uses
   Blade templates with a clean professional design (light gray
   background, white cards, Poppins typography), a collapsible sidebar,
   modals for forms, a toast system and a summary page with statistics
   and calendar, all styled with custom CSS and Tailwind.

   The application includes full authentication (registration, login,
   password recovery, email verification) using Laravel Breeze, CSRF
   protection, authorization middleware, and user profile management.
   It does not implement REST APIs, queues, roles, or automated testing,
   opting instead for a simple and straightforward server-rendered
   approach. The data flow follows the standard cycle: HTTP request →
   controller → validation → model (MongoDB) → notification (if
   applicable) → response (Blade view or redirect). The project is
   ready to deploy with integrated configuration and development
   scripts in Composer.

3. DESCRIPCIÓN DEL DISEÑO

3.1 Modelado de la Lógica de Negocio

   3.1.1 Análisis del Flujo de Procesos del Negocio

   El flujo principal del sistema comienza con el registro e inicio de
   sesión del usuario. Una vez autenticado, el usuario accede al
   dashboard donde visualiza sus tareas y puede crear nuevas. Cada tarea
   pasa por una máquina de estados: al crearse queda en estado
   "pendiente". El sistema evalúa continuamente la fecha y hora de cada
   tarea contra la hora actual: si faltan menos de 72 horas se considera
   "próxima a vencer", y si la fecha ya pasó se marca como "vencida".
   El usuario puede marcar manualmente una tarea como "completada".

   El sistema de notificaciones opera en dos modos:
   - Notificaciones de estado: se generan automáticamente cuando una
     tarea entra en estado "pendiente" (próxima a vencer) o "vencida".
   - Notificaciones de acción: se generan cuando el usuario realiza
     acciones como crear, actualizar, completar o eliminar una tarea.

   3.1.2 Estructuración de las Reglas y Datos del Negocio

   Reglas de negocio:

   R1: Toda tarea debe tener título, descripción, fecha y hora
        obligatorios.
   R2: El estado inicial de una tarea es "pendiente".
   R3: Una tarea se marca como "vencida" automáticamente cuando su
        fecha/hora es anterior a la fecha/hora actual.
   R4: Una tarea "vencida" puede revertirse a "pendiente" si el usuario
        edita su fecha/hora a un valor futuro.
   R5: Una tarea solo puede marcarse como "completada" manualmente por
        el usuario.
   R6: Las notificaciones de estado (pendiente/vencida) se generan una
        sola vez por transición de estado.
   R7: Las notificaciones de estado se eliminan automáticamente al
        cambiar el usuario el estado de la tarea.
   R8: Si el usuario descarta una notificación de estado
        (notification_dismissed_at), no se generan nuevas notificaciones
        para esa tarea.

   Datos del negocio:

    • Usuario: name, email, password, email_verified_at
   • Tarea: title, description, date, time, status, user_id,
            notification_dismissed_at, last_state_notified
   • Notificación: user_id, task_id, type, title, message, read

3.2 Diseño de la Arquitectura Lógica del Sistema

   El sistema sigue el patrón MVC (Modelo-Vista-Controlador) de Laravel:

   • Modelos (App\Models):
     - User: autenticación, relación con tareas y notificaciones
     - Task: gestión de tareas, relación con usuario y notificaciones
     - Notification: gestión de notificaciones, relación con usuario y
       tarea

    • Controladores (App\Http\Controllers):
      - Auth/ (9 controladores): login, registro, contraseña,
        verificación de email (Laravel Breeze)
      - TaskController: CRUD de tareas, sincronización de estados y
        notificaciones
      - NotificationController: listado, filtrado, marcado y eliminación
        de notificaciones
      - ProfileController: gestión de perfil de usuario (Breeze)

    • Peticiones (App\Http\Requests):
      - ProfileUpdateRequest: validación de name y email
      - Auth/LoginRequest: validación de login con rate limiting

    • Componentes Blade (resources/views/components/):
      - application-logo, auth-session-status, input-error, input-label,
        primary-button, text-input

    • Componente de Vista (App\View\Components):
      - GuestLayout: layout base para páginas de invitado

    • Vistas (resources/views):
      - dashboard.blade.php: panel principal con lista de tareas
      - home.blade.php: resumen con estadísticas, calendario (Alpine.js)
      - notificaciones.blade.php: panel de notificaciones con filtros
      - layouts/guest.blade.php: layout base para auth (Breeze)
      - auth/: login, register, verify-email, forgot-password,
        reset-password, confirm-password
      - profile/: edit + partials (update-profile, update-password,
        delete-user)

    • Comandos Artisan:
      - mongo:setup-indexes: crea índices en MongoDB (email único,
        user_id, compuestos)

   • Base de datos:
     - MongoDB con tres colecciones: users, tasks, notifications
     - Conexión NoSQL sin esquema rígido (schema-less)

    • Enrutamiento:
      Routes/web.php:
      - GET / -> Redirección a /dashboard
      - GET /dashboard - Dashboard principal con lista de tareas
      - GET /home - Resumen con estadísticas, calendario y próxima
        tarea
      - POST /tasks - Crear tarea
      - PUT/PATCH /tasks/{task} - Actualizar tarea
      - DELETE /tasks/{task} - Eliminar tarea
      - GET /notificaciones - Listar notificaciones (con filtro)
      - PATCH /notificaciones/{id}/read - Marcar como leída
      - PATCH /notificaciones/read-all - Marcar todas como leídas
      - DELETE /notificaciones/{id} - Eliminar notificación
      - DELETE /notificaciones - Limpiar todas
      - GET /profile - Editar perfil
      - PATCH /profile - Actualizar perfil
      - DELETE /profile - Eliminar cuenta

      Routes/auth.php (Breeze):
      - GET/POST /register, /login, /forgot-password, /reset-password,
        /confirm-password, /logout
      - GET /verify-email, /verify-email/{id}/{hash}
      - POST /email/verification-notification
      - PUT /password
      - GET /logout (redirección GET para cierre de sesión)

   • User 1 -- N Task (un usuario tiene muchas tareas)
   • User 1 -- N Notification (un usuario tiene muchas notificaciones)
   • Task 1 -- N Notification (una tarea tiene muchas notificaciones)

4.3 Diagrama de Secuencia

   Secuencia: Creación de una tarea

   Usuario -> Dashboard: Ingresa al dashboard
   Usuario -> Modal: Abre modal de creación
   Usuario -> Modal: Completa formulario (título, descripción, fecha,
                      hora)
   Modal -> TaskController@store: Envía datos vía POST
   TaskController -> Task: Crea registro en MongoDB
   TaskController -> Notification: Crea notificación de tipo "creada"
   Notification -> TaskController: Retorna notificación creada
   TaskController -> Dashboard: Redirecciona con mensaje de éxito
   Dashboard -> Usuario: Muestra tarea en lista y toast de éxito

   Secuencia: Detección de tarea vencida

   Scheduler -> TaskController@index: Carga dashboard del usuario
   TaskController -> Task: Obtiene tareas del usuario
   TaskController -> syncTaskStateNotification: Evalúa cada tarea
   sync -> Carbon: Calcula diferencia horaria
   alt [diferencia <= 0]
      sync -> Task: Actualiza estado a "vencida"
      sync -> Notification: Crea notificación de tipo "vencida"
   else [diferencia <= 72h]
      sync -> Notification: Crea notificación de tipo "pendiente"
   end
   TaskController -> View: Renderiza dashboard con notificaciones

   [Inicio] --> [Usuario se registra/inicia sesión]

5. MÓDULOS

5.1 Módulo: Autenticación

   5.1.1 Texto Explicativo
   Módulo responsable del registro, inicio de sesión, recuperación de
   contraseña y verificación de correo electrónico. Implementado con
   Laravel Breeze.

   5.1.2 Descripción de la Interfaz
   Formularios de registro y login con validación de campos, tipografía
   Poppins de Google Fonts y footer con iconos de GitHub, animación
   decorativa y copyright. Página de recuperación de contraseña con
   envío de email. Verificación de correo electrónico mediante link
   enviado al email del usuario. Diseño responsive con paneles
   deslizables en móvil.

   5.1.3 Descripción en Lenguaje de Diseño
   Controladores Auth\* proporcionados por Laravel Breeze. Vistas
   ubicadas en resources/views/auth/ con estilos en style.css. La
   página de login incluye footer con iconos de GitHub y gif
   decorativo, tipografía Poppins cargada desde Google Fonts, y
   diseño responsive con paneles deslizables en dispositivos móviles.
   Middleware de autenticación y verificación de email.

   5.1.4 Módulos Utilizados
   Laravel Breeze, Laravel Fortify (subyacente), Mail.

   5.1.5 Organización de los Datos
    Colección users en MongoDB. Campos: _id, name, email, password,
    email_verified_at, remember_token, created_at, updated_at.

   5.1.6 Comentarios
   La autenticación sigue el estándar de Laravel 12 con Breeze,
   adaptado para MongoDB mediante el paquete mongodb/laravel-mongodb.

5.2 Módulo: Gestión de Tareas

   5.2.1 Texto Explicativo
   Módulo principal del sistema que implementa el CRUD completo de
   tareas, incluyendo la máquina de estados y la sincronización de
   notificaciones automáticas.

   5.2.2 Descripción de la Interfaz
   Dashboard con lista de tareas ordenadas por fecha de actualización.
   Modal para crear y editar tareas con campos de título, descripción,
   fecha y hora. Botón para marcar como completada. Indicador visual
   del estado de cada tarea.

   5.2.3 Descripción en Lenguaje de Diseño
   TaskController con métodos: index, store, update, destroy. Métodos
   privados: syncTaskStateNotification, upsertTaskNotification,
   cleanupLegacyDuplicates. Modelo Task con conexión MongoDB y
   colección tasks.

   5.2.4 Módulos Utilizados
   TaskController, Task (Model), Notification (Model), Carbon.

   5.2.5 Organización de los Datos
   Colección tasks en MongoDB. Campos: _id, user_id, title, description,
   date, time, status, notification_dismissed_at, last_state_notified,
   created_at, updated_at.

   5.2.6 Comentarios
   La máquina de estados evalúa automáticamente el vencimiento de cada
   tarea al cargar el dashboard. El método syncTaskStateNotification
   calcula la diferencia horaria y actualiza estados y notificaciones.

5.3 Módulo: Notificaciones

   5.3.1 Texto Explicativo
   Sistema de notificaciones dual que genera alertas automáticas tanto
   por cambios de estado de las tareas como por acciones del usuario.

   5.3.2 Descripción de la Interfaz
   Panel de notificaciones con pestañas de filtro (todas, no leídas,
   leídas). Cada notificación muestra tipo, título, mensaje y fecha.
   Botones para marcar como leída, marcar todas como leídas, eliminar
   individualmente y limpiar todas. Notificaciones de estado
   (pendiente/vencida) se muestran al usuario mediante un toast al
   ingresar al dashboard.

   5.3.3 Descripción en Lenguaje de Diseño
   NotificationController con métodos: index, markAsRead, markAllAsRead,
   destroy, clearAll. Métodos privados: findNotification,
   markRelatedTaskAsDismissed. Modelo Notification con conexión MongoDB
   y colección notifications.

   5.3.4 Módulos Utilizados
   NotificationController, Notification (Model), Task (Model).

   5.3.5 Organización de los Datos
   Colección notifications en MongoDB. Campos: _id, user_id, task_id,
   type, title, message, read, created_at, updated_at. Tipos de
   notificación: creada, actualizada, eliminada, completada, pendiente,
   vencida.

   5.3.6 Comentarios
   Las notificaciones de estado se eliminan automáticamente al cambiar
   el estado de la tarea. Si el usuario descarta una notificación de
   estado, no se generan nuevas notificaciones para esa tarea
   (controlado por notification_dismissed_at).

5.4 Módulo: Perfil de Usuario

   5.4.1 Texto Explicativo
   Módulo que permite al usuario gestionar su información personal y
   configuración de la cuenta.

   5.4.2 Descripción de la Interfaz
    Página de perfil con diseño de tarjetas blancas, avatar con inicial
    del usuario, formularios para editar nombre y email, cambio de
    contraseña, y opción de eliminar cuenta con modal de confirmación.
    Botones primary (azul #2563eb) y danger (outline rojo).

   5.4.3 Descripción en Lenguaje de Diseño
   ProfileController proporcionado por Laravel Breeze con métodos edit,
   update, destroy. Vistas en resources/views/profile/.

   5.4.4 Módulos Utilizados
   ProfileController, User (Model), Laravel Breeze.

   5.4.5 Organización de los Datos
   Colección users en MongoDB. Actualización de campos name, email,
   password.

   5.4.6 Comentarios
   La zona horaria se define globalmente en config/app.php (America/Lima).
   El controlador TaskController accede a auth()->user()->timezone con
   fallback a config('app.timezone'), pero no hay UI para que el usuario
   la cambie. La eliminación de cuenta elimina en cascada las tareas y
   notificaciones del usuario.

5.5 Módulo: Dashboard y Resumen

   5.5.1 Texto Explicativo
   Módulo que proporciona dos vistas complementarias: el dashboard
   principal con la lista completa de tareas y la página de resumen
   (/home) con estadísticas visuales, calendario y próxima tarea.

   5.5.2 Descripción de la Interfaz
   Vista /dashboard con lista completa de tareas ordenadas por fecha
   de actualización, tarjetas de resumen rápido (total, próximas,
   última editada), y toast de bienvenida con alerta de tareas
   pendientes o vencidas.

   Vista /home con diseño de cuadrícula de dos columnas que incluye
   tarjetas de estadísticas (total, completadas, pendientes, vencidas),
   barra de progreso circular con porcentaje, calendario mensual
   interactivo con tareas marcadas por estado, lista de próximas tareas
   pendientes, y acceso directo a las secciones del sistema.

   5.5.3 Descripción en Lenguaje de Diseño
   TaskController@index para dashboard con lógica de sincronización
   de estados y notificaciones. Ruta /home con lógica inline en web.php
   que calcula estadísticas (total, completadas, pendientes, vencidas,
   porcentaje de progreso), determina la próxima tarea pendiente
   ordenada por fecha y hora, genera datos del calendario (mes, año,
   nombre del mes), y serializa las tareas en JSON para el calendario
   interactivo. Estilos definidos en style2.css (dashboard) y
   style_home.css (resumen). La vista home carga Google Fonts (Poppins)
   y Font Awesome con carga diferida.

   5.5.4 Módulos Utilizados
   TaskController, Task (Model), Notification (Model), Carbon.

   5.5.5 Organización de los Datos
   Consultas a colección tasks filtradas por user_id. Cálculo de
   estadísticas mediante conteo por estado. Serialización de tareas
   en JSON (title, date, time, status) para el calendario. Conteo
   de notificaciones no leídas para el badge del sidebar.

   5.5.6 Comentarios
   El dashboard es la vista principal a la que se redirecciona al
   usuario tras iniciar sesión. La vista /home se añadió posteriormente
   como mejora para proporcionar un resumen visual más completo con
   calendario y estadísticas agregadas. El sidebar incluye un nuevo
   ítem "Resumen" con icono de casa en todas las vistas del sistema.

6.6 Casos de Prueba (Obligatorio)

   CP-01: Registro de usuario exitoso
      Entrada: nombre, email, contraseña válidos
      Resultado esperado: cuenta creada, redirección a dashboard

   CP-02: Inicio de sesión con credenciales inválidas
      Entrada: email no registrado, contraseña incorrecta
      Resultado esperado: mensaje de error, no autenticado

   CP-03: Creación de tarea con datos válidos
      Entrada: título, descripción, fecha futura, hora
      Resultado esperado: tarea creada en estado "pendiente",
      notificación "creada" generada

   CP-04: Vencimiento automático de tarea
      Precondición: tarea con fecha pasada
      Resultado esperado: estado cambiado a "vencida",
      notificación "vencida" generada

   CP-05: Completar tarea
      Precondición: tarea en estado "pendiente" o "vencida"
      Resultado esperado: estado cambiado a "completed",
      notificación "completada" generada

   CP-06: Editar tarea vencida con fecha futura
      Precondición: tarea en estado "vencida"
      Entrada: fecha futura
      Resultado esperado: estado revertido a "pendiente"

   CP-07: Eliminar tarea
      Precondición: tarea existente
      Resultado esperado: tarea eliminada, notificaciones relacionadas
      eliminadas, notificación "eliminada" generada

   CP-08: Marcar notificación como leída
      Precondición: notificación no leída
      Resultado esperado: notificación marcada como leída

   CP-09: Filtro de notificaciones
      Entrada: filtro "no leídas"
      Resultado esperado: solo se muestran notificaciones con read=false

   CP-10: Eliminación de cuenta
      Precondición: usuario autenticado
      Resultado esperado: cuenta, tareas y notificaciones eliminadas

7. CONCLUSIONES Y RECOMENDACIONES

7.1 Conclusiones

   • TaskFlow cumple con todos los objetivos planteados, proporcionando
     una solución funcional y completa para la gestión de tareas
     personales.

   • La arquitectura MVC con Laravel y MongoDB demostró ser adecuada
     para el proyecto, permitiendo un desarrollo ágil y mantenible.

   • El sistema de notificaciones inteligentes implementado
     exitosamente proporciona valor añadido al usuario al mantenerlo
     informado sobre el estado de sus tareas sin intervención manual.

   • La interfaz de usuario responsive y los componentes interactivos
     (modales, toast, sidebar) ofrecen una experiencia de usuario
     moderna y agradable.

   • La integración de Laravel Breeze para autenticación proporcionó
     una base segura y probada para la gestión de usuarios.

   • El uso de MongoDB como base de datos NoSQL permitió flexibilidad
     en el modelo de datos y facilitó la implementación de relaciones
     entre usuarios, tareas y notificaciones.

7.2 Recomendaciones

   1. Implementar pruebas automatizadas (unitarias y de integración)
      para garantizar la calidad del código y facilitar la detección
      temprana de errores.

   2. Incorporar procesamiento asíncrono mediante colas (Laravel Queues)
      para la generación y envío de notificaciones, mejorando el
      rendimiento del sistema.

   3. Desarrollar una API REST pública para permitir integraciones con
      aplicaciones de terceros y clientes móviles.

   4. Agregar gestión avanzada de roles y permisos para soportar
      múltiples tipos de usuarios.

   5. Implementar monitoreo, registros (Logging) y procesos CI/CD para
      facilitar el despliegue continuo y la depuración en producción.

   6. Integrar subida de archivos (adjuntos) a las tareas para
      enriquecer la información gestionada.

   7. Implementar modo offline y funcionalidad PWA para mejorar la
      accesibilidad desde dispositivos móviles.

   8. Agregar soporte multi-idioma para ampliar el alcance del sistema
      a usuarios de diferentes regiones.

9.2 Código de Creación de BD y Tablas

   MongoDB no utiliza esquemas rígidos ni tablas. Las colecciones se
   crean automáticamente al insertar el primer documento. La conexión
   se configura en config/database.php:

    'mongodb' => [
        'driver'   => 'mongodb',
        'dsn'      => env('MONGODB_URI', 'mongodb://127.0.0.1:27017'),
        'database' => env('MONGODB_DATABASE', 'taskflow'),
    ],

   Los modelos extienden MongoDB\Laravel\Eloquent\Model y definen la
   colección correspondiente mediante la propiedad $collection.
